Case study

OptiCube Games Website

HTML • CSS • Vanilla JavaScript • GitHub Pages (GitHub Actions) • Custom domain

A production static marketing website for a game studio, including a dedicated game page and a full press kit. The focus was engineering clarity: maintainable structure, resilient loading, accessible interactions, and a UI that stays understandable on any screen size.

Problem/Goals

  • Ship a production-quality marketing site without a backend, while keeping updates low-friction.
  • Make navigation and interactive elements work reliably on mobile and with keyboard-only input.
  • Present press-ready media (factsheet, trailer, screenshots) in a structured, easy-to-browse format.
  • Avoid fragile UI: validate data inputs (media JSON) and degrade gracefully on failures.

My Role

I owned the frontend implementation and the engineering decisions behind the site's structure, interactions, and deployment. I built the reusable layout system (partials), interactive navigation, and the press kit media experiences (carousel, data-driven library, and lightbox).

Architecture

  • Static-first: plain HTML/CSS/JS for predictable hosting, simple debugging, and low maintenance.
  • Shared layout via partials: a runtime partial loader pulls the header/footer from shared HTML fragments to keep one source of truth across pages.
  • Data-driven UI: press kit visual library is backed by JSON categories, rendered with small DOM-focused utilities.
  • Failure-aware: validate JSON shape before rendering; if data fails, show a usable fallback instead of crashing the page.

Key Implementations

  • Runtime partial loading: reduced duplication and made global navigation updates safe and fast.
  • Mobile navigation behavior: correct ARIA expanded state, close on outside click, and Escape key support.
  • Press kit media hub: factsheet + trailer embed + screenshots carousel with thumbnail navigation.
  • Visual library: JSON-backed categories with tabs and a lightbox viewer for browsing assets.
  • Resilient loading: input validation for media JSON and graceful fallbacks when data is missing or invalid.
  • Scroll effect: a smooth scroll-based visual effect using a CSS custom property updated via requestAnimationFrame.

Accessibility & UX

  • Kept interactive controls discoverable and consistent across pages.
  • Used ARIA state for mobile navigation and supported keyboard escape patterns.
  • Designed press kit navigation around quick scanning: thumbnails, tabs, and clear hierarchy.
  • Focused on predictable layout and readable typography across desktop/tablet/mobile.

Performance

I avoided claiming performance wins without measurement. If you want, I can add a short “measured results” block once you run Lighthouse or share numbers.

  • What I'd measure: Lighthouse (Perf/SEO/Best Practices/A11y), LCP/CLS/INP, total JS/CSS bytes, and image sizes by page.
  • Scroll effect hygiene: update a single CSS custom property from requestAnimationFrame to keep scroll work predictable.
  • Static baseline: no framework runtime and a simple dependency surface.

Deployment

  • Deployed via GitHub Pages with a GitHub Actions workflow.
  • Configured a custom domain and kept the hosting pipeline simple and repeatable.
  • Optimized for safe updates: static content + deterministic deploy = fewer production surprises.

Tradeoffs

  • Runtime partial loading depends on JavaScript; the benefit is zero duplication, the cost is an extra request and a bit more client-side logic.
  • JSON-driven UI needs validation and fallbacks; the upside is maintainability and quick content updates.
  • GitHub Pages keeps infra simple, but limits server-side features; everything must be designed to run client-side.

What I'd Do Next

  • Add Playwright smoke tests for navigation, carousel, tabs, and lightbox.
  • Introduce lightweight schema validation utilities for JSON data (and error reporting).
  • Set explicit performance budgets and track regressions with Lighthouse CI.
  • Add a small build step for asset optimization (image resizing, preloading, and cache-friendly filenames).

Stripe Alignment

I approach static sites like production systems: minimize sharp edges, keep abstractions small, and make the failure modes boring. On OptiCube, that meant a maintainable shared layout, data-driven rendering with validation and graceful fallbacks, accessible interaction patterns (ARIA + keyboard support), and a performance mindset grounded in measurement. The goal is a site that's easy to ship, easy to change, and hard to break.
